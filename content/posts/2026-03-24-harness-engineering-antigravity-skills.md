AI 에이전트가 단순히 '똑똑한 대화 상대'를 넘어 '신뢰할 수 있는 도구'가 되기 위해서는 강력한 제어 장치가 필요합니다. 이번 포스트에서는 하니스 엔지니어링(Harness Engineering)의 개념을 통해 안티그래비티(Antigravity) 환경에서 클로드 스킬과 규칙을 어떻게 설계하고 적용하는지 심층적으로 다룹니다.

## Overview

**하니스 엔지니어링(Harness Engineering)**이란 AI 모델이 제멋대로 힘을 흩뜨리지 않고, 우리가 의도한 방향으로만 움직이게 만드는 '마구(Harness)'를 설계하는 일입니다. 안티그래비티 환경에서 이는 단순한 설정 이상의 의미를 갖습니다. 에이전트 매니저, 에디터, 브라우저가 유기적으로 작동하도록 **룰(Rules), 스킬(Skills), MCP(Model Context Protocol), 워크플로우(Workflows)**라는 네 가지 핵심 요소를 배치하여 최적화된 작업 환경을 구축하는 과정입니다.

## Background / Problem

단순히 모델에게 프롬프트만 던지는 방식은 몇 가지 고질적인 문제를 야기합니다:
1.  **일관성 부족**: 매번 같은 작업을 시켜도 결과물의 스타일이나 품질이 들쭉날쭉합니다.
2.  **컨텍스트 오버헤드**: 모든 지침과 지식을 대화창에 직접 입력하면 컨텍스트 길이가 늘어나 비용이 상승하고 모델의 집중력이 분산됩니다.
3.  **제어 불능**: 에이전트가 어떤 문서를 참고해야 할지, 어디까지 수정 권한을 가질지 명확한 가이드라인이 없으면 위험한 코드를 실행하거나 엉뚱한 결과물을 낼 수 있습니다.

따라서 에이전트가 작업에 몰입할 수 있도록 '바깥 틀(Harness)'을 씌워주는 설계가 필수적입니다.

## Solution / Implementation

안티그래비티의 하니스 구조는 다음과 같이 계층화된 제어 시스템을 제공합니다.

### 1. 계층적 규칙 설계 (Rules)
규칙은 에이전트가 항상 지켜야 하는 헌법과 같습니다.
-   **글로벌 규칙 (`~/.gemini.md`)**: 코딩 스타일(예: "항상 Python은 UV를 사용해라"), 시간대, 로깅 습관 등 모든 프로젝트에 공통 적용되는 지침입니다.
-   **로컬 규칙 (`.agents/rules/`)**: 특정 워크스페이스에서만 유효한 규칙입니다. 시각화 라이브러리 선택(Matplotlib vs Plotly)이나 프로젝트 전용 용어집 등을 정의합니다.
-   **인라인 규칙**: 대화 중 "한국어로만 답변해줘"와 같이 즉각적으로 수정이 필요한 지침입니다.

### 2. 전문 지식의 패키징 (Skills)
스킬은 재사용 가능한 전문 지식의 묶음입니다. 스킬의 핵심은 **프로그레시브 디스클로저(Progressive Disclosure)**입니다.
-   안티그래비티는 스킬 폴더의 `YAML Front-matter`에 포함된 `description`만 먼저 읽습니다.
-   필요한 시점에만 전체 내용을 로드하므로, MCP 방식보다 **토큰 사용량을 획기적으로 절감**하면서도 지연 시간을 최소화합니다.
-   `Skill Creator`를 활용하면 기존 워크플로우를 영어 기반의 프로페셔널 스킬로 자동 변환하여 성능을 극대화할 수 있습니다.

### 3. 외부 연결과 자동화 (MCP & Workflows)
-   **MCP**: GitHub, Database, 웹 API 등 외부 도구와의 연결 통로입니다. `GitHub MCP`를 통해 에이전트가 직접 Issue를 만들고 PR을 보내도록 위임할 수 있습니다.
-   **Workflows**: 슬래시(`/`) 명령어로 복잡한 다단계 작업을 자동화합니다. "데이터 수집 -> 분석 -> 리포트 생성 -> PPTX 변환" 과정을 하나의 스킬처럼 트리거합니다.

### 4. 물리적 하니스 구성 (Path Management)
스킬과 규칙이 실제로 동작하게 하려면 정확한 경로 설정이 중요합니다.
-   **글로벌 경로**: 맥 기준 `~/.gemini/antigravity/` 아래에 `skills/` 폴더를 생성하고 스킬 파일(`.md`)을 넣습니다.
-   **로컬 경로**: 프로젝트 루트의 `.agents/skills/` 또는 `.agents/rules/` 폴더를 사용합니다.

## Key Takeaways

1.  **모델보다 하니스**: 어떤 모델(Gemini, Claude, GPT)을 쓰느냐보다, 그 모델을 감싸는 **스킬과 룰의 설계(Harness)**가 결과물의 퀄리티에 더 큰 영항을 미칩니다.
2.  **경량화 전략**: 모든 지식을 룰에 넣기보다, 특정 목적의 스킬로 분리하여 필요할 때만 호출되게 하는 것이 비용과 성능 측면에서 유리합니다.
3.  **전문가 노하우의 자산화**: 잘 설계된 스킬 파일 하나가 숙련된 에이전트 하나를 복제하는 것과 같습니다.

## Gachas

-   **경로 명칭의 혼선**: 문서나 버전에 따라 `.agents`인지 `.agent`인지, 혹은 폴더명 뒤에 `s`가 붙는지(`rules` vs `rule`) 헷갈릴 수 있습니다. 안티그래비티 공식 사양에 따르면 **`rules`, `skills`, `workflows`와 같이 복수형(s)**을 사용하는 것이 표준입니다.
-   **토큰 폭식 (MCP 주의)**: 컨텍스트를 과도하게 가져오는 MCP(예: Context24 등)를 남용하면 토큰 사용량이 급증합니다. 지식 베이스는 가급적 스킬(`Progressive Disclosure`) 형태로 관리하는 것이 경제적입니다.
-   **언어의 장벽**: 스킬의 구현 내용은 영어로 작성할 때 모델이 훨씬 정교하게 이해하고 실행합니다. 설정은 한국어로 하더라도 실제 로직은 영어를 권장합니다.
